# Domain Theses v1

These are the stable thesis primitives used by the Companion knowledge layer.

- T1 — The Inversion Thesis: GRC should work bottom-up from control reality, not top-down from frameworks.
- T2 — Architecture Before Automation: you cannot automate a broken process; design information architecture first.
- T3 — GRC Is a Product: GRC should run like a product org with customers, iteration, and feedback loops.
- T4 — Engineering Is a Capability, Not a Role: GRC Engineering is a stack of practices accessible to anyone.
- T5 — Outcomes Over Activity: measure security outcomes and decision quality, not compliance activity metrics.
- T6 — Data Layer as Foundation: GRC is fundamentally a data engineering problem.
- T7 — The Human Interface Imperative: human interfaces must be engineered with the same rigor as system interfaces.
- T8 — Scaffolding Over Models: AI success depends on workflow scaffolding, not model selection.
- T9 — Commoditization Escalator: each GRC capability layer commoditizes in turn; move up or get automated.
- T10 — Invisible Integration: the best GRC design is invisible to engineers; compliance becomes a byproduct.
- T11 — Leverage, Don't Rebuild: integrate with existing infrastructure rather than building parallel systems.
- T12 — Vendor Neutrality as Authority: credibility comes from independence, not vendor affiliation.

Legacy phrasing preserved:

- GRC engineering is the work of making governance, risk, and compliance function like a real operating system.
- Compliance should describe and verify good work, not replace it.
- Audit evidence is healthiest when produced by normal system activity.
- The best GRC work reduces translation loss between executives, engineers, auditors, legal, sales, and security.
- Frameworks are maps. The organisation is the terrain.
- Tools amplify operating models. They do not rescue broken ones.
- A useful control changes or verifies real behaviour.
- A learning artefact is more valuable than a memorised definition.
